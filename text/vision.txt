## SCHEMA / DATA MODEL (UPDATE)

IM-006 Region Shape System
-- Regions use MULTIPOLYGON as canonical shape
-- Format:

simulation:
spatial:
type: region
shape:
type: multipolygon
polygons:
- [[x1, y1], [x2, y2], [x3, y3]]
- [[x4, y4], [x5, y5], [x6, y6]]

-- Single polygon = multipolygon with one entry
-- Coordinates are in world space (MapSimulation)
-- Polygons should remain low resolution (10–50 points)
-- Holes are NOT supported (future extension)

IM-007 Image Asset System
-- Images are stored as ENTITY ASSETS, not renderer properties
-- Images are reusable across:
--- map rendering
--- UI (entry cards)
--- inspection views

-- Format:

assets:
image:
path: "assets/images/region_001.png"
pixel_size: [2048, 2048]

IM-008 Image Scaling Model
-- Images are NOT defined by fixed world_size
-- Images scale based on DETAIL LEVEL

-- Format:

assets:
image:
path: "..."
pixel_size: [...]
detail_level: "planet | region | city | building"

-- Renderer selects appropriate representation based on zoom

IM-009 Image Placement
-- Images are anchored to the REGION BOUNDING BOX
-- Polygon defines logical region
-- Image is scaled to bounding box automatically
-- No manual transform required (for now)

IM-010 Logic vs Rendering Separation
-- Simulation logic MUST NOT depend on assets
-- Assets (images) are purely visual
-- Polygon shapes are used for:
--- containment
--- adjacency (future)
--- simulation logic

-- Rendering layer uses assets independently of simulation logic

SCHEMA / DATA MODEL (UPDATE)

IM-011 Culture Spatial Model
-- Culture has NO fixed spatial definition
-- Culture presence is derived from population distribution

-- Pops define culture:
pop:
culture: culture_id
position: [x, y]

-- Regions store aggregated culture distribution (cached)
-- Culture representation is built bottom-up:
--- pop → region → higher-level aggregations

-- Enables:
--- dynamic spread via migration
--- culture intensity (% based)
--- simulation-driven evolution

IM-012 Structure Spatial Model
-- Structures use hybrid spatial representation:
--- point anchor (required)
--- optional polygon footprint

-- Format:

simulation:
spatial:
type: structure
position: [x, y]
shape:
type: multipolygon
polygons:
- [[...], [...], [...]] # optional

-- Position = interaction/reference point
-- Shape = physical footprint (if applicable)

IM-013 Planet Multi-Representation System
-- Planets support multiple simulation representations

-- Format:

representations:
space:
orbit:
parent: ...
a: ...
e: ...
map:
surface_ref: "planet_surface_id"

-- SpaceSimulation uses orbital representation
-- MapSimulation uses surface_ref

-- Enables:
--- seamless transition between space and surface
--- multi-layer simulation (space ↔ region ↔ local)

IM-015 Latent NPC Generation
-- Implicit pop members are NOT instantiated entities

-- They become entities only when:
--- player encounters them
--- system requires them (events, UI, simulation detail)

-- Enables scalable population without full simulation

IM-016 Entity Card System
-- Every entity is represented as a "card"

entity:
id: entity_id

knowledge:
state: untouched | seen | known | detailed

-- Knowledge state controls visible information

IM-017 Discovery System (Touched / Untouched)
-- Entities start as "untouched"
-- Player interaction reveals them progressively

States:
--- untouched → no access
--- seen → basic info
--- known → partial attributes
--- detailed → full data

-- Discovery is persistent across sessions

IM-018 Deck-Based World Seeding

-- Before simulation start:
--- player selects cards
--- cards modify world generation

-- Effects may include:
--- spawning entities
--- modifying regions

IM-019 Heatmap System (Spatial Analysis Layer)
-- Heatmaps represent spatial aggregation of simulation data

heatmap:
id: heatmap_id

source:
type: entity_type
attribute: attribute_name

aggregation:
type: spatial_sum | density | average

-- Heatmaps are:
--- derived at runtime
--- resolution-dependent (based on zoom)
--- not permanently stored (optional caching)

IM-020 Ledger System (Statistical Aggregation Layer)
-- Ledger provides structured, non-spatial summaries

ledger:
id: ledger_id

source:
type: entity_type

metrics:
- metric_name_1
- metric_name_2

-- Ledgers aggregate:
--- totals
--- distributions
--- categorized counts

IM-021 Analysis Layer (Separation of Concerns)
-- Heatmap and Ledger belong to a dedicated analysis layer

Location:
sim_core/analysis/

-- Responsibilities:
--- collect data from simulations
--- compute derived values
--- provide structured outputs for UI

-- Simulations MUST NOT contain aggregation logic
